# Step 2: 제품 요구사항 정의서 (PRD)

## 단계 정보
- **단계**: 2/10 - 제품 요구사항 정의서
- **스킬**: `dev-product_manager`
- **입력**: `docs/requirements.md`
- **산출물**: `docs/prd.md`

---

## 실행 단계

### 1. 컨텍스트 로드

다음 파일을 읽고 분석합니다:
- `docs/requirements.md` - 요구사항 정의서

### 2. 스킬 로드

`dev-product_manager` 스킬을 로드하여 제품 관리 전문 지식을 활용합니다.

### 3. 사용자 스토리 작성

요구사항을 사용자 스토리(User Stories)로 변환합니다:

```
[역할]로서,
[기능]을 원한다,
[가치/목적]을 위해서.
```

**예시**:
```
US-001: 사용자 가입
신규 사용자로서,
계정을 등록하고 싶다,
시스템 기능을 사용하기 위해서.

수락 기준:
- Given 사용자가 가입 페이지에 있음
- When 이메일, 비밀번호를 입력하고 제출함
- Then 계정이 생성되고 인증 메일이 발송됨
```

### 4. 기능 우선순위

MoSCoW 방법을 사용하여 분류합니다:

| 우선순위 | 설명 | 기능 |
|--------|------|------|
| **Must** | 반드시 있어야 함 | 핵심 기능 |
| **Should** | 있으면 좋음 | 중요 기능 |
| **Could** | 나중에 추가 가능 | 부가적인 기능 |
| **Won't** | 이번에는 하지 않음 | 향후 고려 항목 |

### 5. 수락 기준

각 기능에 대해 수락 기준(Acceptance Criteria)을 정의합니다:

```gherkin
Feature: 사용자 로그인

  Scenario: 로그인 성공
    Given 사용자가 이미 가입된 상태임
    And 사용자가 로그인 페이지에 있음
    When 올바른 이메일과 비밀번호를 입력함
    And 로그인 버튼을 클릭함
    Then 메인 페이지로 이동함
    And 환영 메시지를 표시함

  Scenario: 비밀번호 오류
    Given 사용자가 이미 가입된 상태임
    And 사용자가 로그인 페이지에 있음
    When 올바른 이메일이지만 잘못된 비밀번호를 입력함
    And 로그인 버튼을 클릭함
    Then 오류 메시지 "비밀번호 오류"를 표시함
    And 로그인 페이지를 유지함
```

### 6. 문서 생성

`docs/prd.md`를 생성합니다:

```markdown
# {프로젝트 이름} - 제품 요구사항 정의서 (PRD)

## 문서 정보
- 버전: 1.0
- 작성자: {작성자}
- 날짜: {날짜}
- 상태: 초안(Draft) / 검토(Review) / 승인(Approved)

## 1. 제품 개요
### 1.1 제품 비전
### 1.2 대상 사용자
### 1.3 핵심 가치

## 2. 사용자 스토리
### 2.1 Epic 리스트
### 2.2 사용자 스토리 상세

## 3. 기능 요구사항
### 3.1 Must Have (P0)
### 3.2 Should Have (P1)
### 3.3 Could Have (P2)

## 4. 비기능 요구사항
### 4.1 성능 요구사항
### 4.2 보안 요구사항
### 4.3 호환성 요구사항

## 5. 수락 기준
### 5.1 기능 수락
### 5.2 비기능 수락

## 6. 마일스톤
### 6.1 MVP 범위
### 6.2 V1.0 범위
### 6.3 향후 계획

## 7. 리스크 및 의존성
### 7.1 리스크 리스트
### 7.2 외부 의존성

## 8. 부록
### 8.1 용어 사전
### 8.2 참고 문서
```

### 7. 사용자 확인

PRD 문서를 사용자에게 보여주고 확인을 요청합니다:

```
[C] 확인 - PRD 완벽, 다음 단계로 진행
[E] 편집 - 내용 수정
[A] 추가 - 사용자 스토리 보충
[P] 우선순위 - 우선순위 조정
```
